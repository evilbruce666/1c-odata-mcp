# Подключение и настройка

Пошаговое руководство: как опубликовать OData в 1С, проверить каждый шаг отдельно
и подключить сервер к Claude Desktop / Claude Code. В конце — таблица диагностики
по реальным ошибкам, на которые легко наткнуться.

Подключение состоит из трёх независимых уровней. Проверяйте их **по порядку** —
так сразу видно, где именно проблема, и не приходится гадать.

```
[1] 1С: публикация OData  →  [2] сеть+авторизация  →  [3] клиент Claude
```

---

## Уровень 1. Публикация OData в 1С

Сервер работает с **любой 1С:Предприятие**, у которой опубликован стандартный
интерфейс OData — неважно, развёрнута база на вашем сервере (IIS/Apache), в облаке
1cfresh, у хостинг-провайдера или в Scloud. Где именно лежит база — определяет лишь
адрес OData (Уровень 2).

Независимо от площадки, **список доступных объектов пуст, пока вы не добавите их
вручную** в настройке стандартного интерфейса OData. Без этого сервис вернёт пустоту
(`"value": []`), а обращение к объекту — «сущность не найдена».

### Что сделать

1. Войдите в базу в режиме **«Предприятие»**.
2. **Все функции** → **Обработки** → **«Настройка стандартного интерфейса OData»**.
   - Если «Все функции» не видно: Главное меню (≡) → Сервис → Параметры →
     ✓ «Отображать команду Все функции».
3. Вкладка **«Состав»** → отметьте объекты (галочками) → **Записать / Сохранить**.

### Минимальный состав для этого сервера (Бухгалтерия 3.0)

| Категория | Объекты |
|---|---|
| Справочники | Контрагенты, Номенклатура, Организации, ДоговорыКонтрагентов, Склады, БанковскиеСчета |
| Документы | РеализацияТоваровУслуг, СчётНаОплатуПокупателю, ПоступлениеТоваровУслуг, ПоступлениеНаРасчётныйСчёт, СписаниеСРасчётногоСчёта, ПлатёжноеПоручение, ПриходныйКассовыйОрдер, РасходныйКассовыйОрдер |
| План счетов | **Хозрасчётный** (обязательно — из него считаются дебиторка и остатки) |
| Регистры бухгалтерии | **Хозрасчётный** |

> Добавляйте только то, что реально нужно для отчётов — это третий контур
> безопасности: сервер физически не увидит объект, которого нет в «Составе».

### ✅ Что проверить в настройках 1С

- [ ] Объекты добавлены в «Состав» **и сохранены** (не забыть «Записать»).
- [ ] В «Составе» есть план счетов **Хозрасчётный** и регистр бухгалтерии
      **Хозрасчётный** — без них `get_debtors` / `get_inventory` не работают.

## Уровень 2. Сеть и авторизация

### Адрес OData

Общий вид адреса стандартного интерфейса OData в 1С (одинаков для любой площадки):

```
https://<сервер>/<имя_базы>/odata/standard.odata/
```

**Завершающий слэш обязателен.** Точный адрес зависит от того, как опубликована база:

- **Собственная публикация** (IIS/Apache): `https://ваш-сервер/<имя_публикации>/odata/standard.odata/`
- **Хостинг 1cfresh и подобные**: адрес выдаёт провайдер в настройках сервиса.
- **Scloud** (как пример облачной площадки): `https://terminal.scloud.ru/<кластер>/<имя_базы>/odata/standard.odata/`
  — кластер и имя базы видны в адресной строке/настройках публикации.

### Учётные данные

Авторизация — **Basic Auth** (логин/пароль). Какой именно пользователь принимается,
зависит от площадки и настроек публикации:

- В типовом случае это **пользователь информационной базы 1С** с включённой
  «Аутентификацией 1С:Предприятия» и заданным паролем (вход по ОС/домену для Basic
  не подойдёт).
- На **некоторых облачных площадках** публикация сверяет данные не с пользователем
  внутри базы, а с **учётной записью облачного аккаунта**.

> ⚠️ **Частый подводный камень.** Отдельный пользователь, заведённый только внутри 1С,
> иногда получает **401** — площадка ждёт учётку облачного аккаунта. Если 401 не уходит,
> попробуйте учётные данные аккаунта площадки, под которыми входите в её личный кабинет.
> (На Scloud в практике сработала именно облачная учётка.)

Для безопасности заведите **отдельного пользователя** с доступом лишь к нужным объектам:
если сервер используется только для чтения — дайте права лишь на чтение; для записи —
права на изменение только тех объектов, что действительно нужны.

### ✅ Самопроверка уровня 2 (без кода)

Откройте свой адрес OData в браузере:

```
https://<сервер>/<имя_базы>/odata/standard.odata/?$format=json
```

Браузер спросит логин/пароль. Дальше:

| Что видите | Вывод |
|---|---|
| Просит логин и **не пускает** (401) | Неверная учётка → проверьте пользователя 1С / учётку площадки (см. выше) |
| Пустил, но `"value": []` | Авторизация ОК, но **«Состав OData» пуст** → вернитесь на Уровень 1 |
| Пустил, в `value` список объектов | ✅ Уровни 1–2 пройдены, переходите к Уровню 3 |

## Уровень 3. Подключение к Claude

Есть два способа запуска сервера: **через `npx`** (ничего не клонировать) и **из
исходников** (для разработки или своих правок).

### Вариант А. Через `npx` (рекомендуется)

Ничего ставить заранее не нужно — `npx` сам скачает и запустит пакет. Настройки
передаются через переменные окружения в самом конфиге Claude.

`claude_desktop_config.json` (**macOS:** `~/Library/Application Support/Claude/`,
**Windows:** `%APPDATA%\Claude\`), добавьте в `mcpServers`:

```json
{
  "mcpServers": {
    "1c-odata": {
      "command": "npx",
      "args": ["-y", "1c-odata-mcp"],
      "env": {
        "ODATA_BASE_URL": "https://<сервер>/<имя_базы>/odata/standard.odata/",
        "ODATA_USERNAME": "...",
        "ODATA_PASSWORD": "..."
      }
    }
  }
}
```

> Минус варианта А: пароль хранится в самом конфиге. Если это нежелательно — вариант Б
> с `.env`. Несколько баз / запись — задаются теми же переменными окружения
> (см. [Несколько баз](#несколько-баз-1с) и [Включение записи](#включение-записи-опционально)).

### Вариант Б. Из исходников (с `.env`, пароль не в конфиге)

```bash
git clone https://github.com/evilbruce666/1c-odata-mcp.git
cd 1c-odata-mcp
npm install
npm run build
cp .env.example .env      # заполните ODATA_BASE_URL / USERNAME / PASSWORD
```

Конфиг Claude — сервер берёт переменные из `.env` через `--env-file`:

```json
{
  "mcpServers": {
    "1c-odata": {
      "command": "/opt/homebrew/opt/node@22/bin/node",
      "args": [
        "--env-file=/Users/<вы>/Projects/1c-odata-mcp/.env",
        "/Users/<вы>/Projects/1c-odata-mcp/dist/index.js"
      ]
    }
  }
}
```

Тонкости варианта Б (именно из-за них бывает «непросто»):

- **Абсолютный путь к `node`.** GUI-приложение Claude Desktop может не видеть `node`
  из вашего PATH. Укажите полный путь (`which node`). На Apple Silicon с Homebrew это
  обычно `/opt/homebrew/opt/node@22/bin/node`.
- **Абсолютные пути** к `.env` и `dist/index.js` (не относительные, не `~`).
- **`--env-file` требует Node 20.6+** (`node -v`).

### Проверка без Claude (для варианта Б, рекомендуется)

probe-скрипты берут переменные из `.env` и секретов не печатают:

```bash
node --env-file=.env dist/scripts/probe-metadata.js   # карта объектов + сверка имён
node --env-file=.env dist/scripts/probe-analytics.js  # дебиторка / остатки
node --env-file=.env dist/scripts/probe-mcp.js        # end-to-end по протоколу MCP
```

Если `probe-metadata` печатает версию OData и сотни объектов — всё готово.

### Подключение к Claude Code

```bash
# через npx
claude mcp add 1c-odata -s user --env ODATA_BASE_URL=… --env ODATA_USERNAME=… --env ODATA_PASSWORD=… -- npx -y 1c-odata-mcp
# или из исходников
claude mcp add 1c-odata -s user -- /opt/homebrew/opt/node@22/bin/node \
  --env-file=/Users/<вы>/Projects/1c-odata-mcp/.env \
  /Users/<вы>/Projects/1c-odata-mcp/dist/index.js
```

Проверка: `claude mcp list`.

### ✅ Что проверить в Claude

- [ ] **Полностью перезапустите Claude Desktop** (Cmd+Q, не просто окно) — иначе
      сервер не подхватится. В списке инструментов появится `1c-odata`.
- [ ] Запрос «проверь соединение с 1С» → `health_check` возвращает версию OData и
      число объектов.
- [ ] Запрос «покажи дебиторку» → список должников.

## Таблица диагностики

| Симптом | Где | Причина | Что делать |
|---|---|---|---|
| **401 Unauthorized** | браузер/curl/health_check | Неверная учётка / не та аутентификация | Проверить «Аутентификация 1С:Предприятия» и пароль; на облачных площадках — попробовать учётку аккаунта площадки |
| `"value": []`, объектов 0 | health_check / probe | «Состав OData» пуст | Уровень 1: добавить объекты и **сохранить** |
| **404 / «сущность не найдена»** | конкретный инструмент | Объект не добавлен в «Состав» | Добавить нужный объект в «Состав OData» |
| **400 Bad Request** на фильтрах | свой код к OData | 1С не декодирует `+` в пробел в `$filter` | Кодировать пробел как `%20` (`encodeURIComponent`), **не** `URLSearchParams` — в этом сервере уже исправлено |
| `get_debtors`/`get_inventory` пустые/ошибка | эти инструменты | Не опубликованы план счетов и/или регистр **Хозрасчётный** | Добавить оба «Хозрасчётный» в «Состав» |
| Инструменты не появились в Claude | Claude Desktop | Не полный перезапуск / `node` не найден / относительные пути | Cmd+Q и заново; абсолютный путь к `node`; абсолютные пути в `args` |
| `Cannot find module` при старте | запуск | Не выполнен `npm run build` | `npm run build`, путь должен указывать на `dist/index.js` |
| Сервер «молчит» / падает сразу | запуск | Невалидный `.env` | Запустить `node --env-file=.env dist/scripts/probe-metadata.js` — он покажет понятную ошибку конфига |

## Несколько баз 1С

Один сервер может обслуживать несколько баз. Опишите каждую в `.env` тройкой
`ODATA_DB_<ИМЯ>_BASE_URL / _USERNAME / _PASSWORD` (+ необязательный `_LABEL`),
при желании задайте `ODATA_DEFAULT_DB`. Полный пример — в `.env.example`.

- Каждый Уровень 1–2 (публикация + авторизация) проверяется для **каждой** базы
  отдельно — у разных баз могут быть разные учётки и разный «Состав OData».
- `<ИМЯ>` из переменной — это значение параметра `database` инструментов;
  список доступных баз отдаёт инструмент **`list_databases`**.
- Старый формат с одной базой (`ODATA_BASE_URL`) продолжает работать (база `default`).

> Несколько **организаций (юрлиц) в одной базе** — это не «несколько баз»: подключение
> одно, фильтр идёт параметром `organization` (список — `list_organizations`).

## Включение записи (опционально)

По умолчанию сервер работает **только на чтение**. Чтобы разрешить создание/изменение
данных, снимите **два** независимых предохранителя:

1. Глобально: `READ_ONLY=false` в `.env`.
2. На нужной базе: `ODATA_DB_<ИМЯ>_WRITABLE=true` (базы без флага остаются read-only —
   можно открыть запись в одну базу и защитить остальные).

Дополнительно:
- у OData-пользователя в 1С должны быть **права на запись** нужных объектов;
- пишущие инструменты по умолчанию работают в режиме **предпросмотра** (dry-run) и
  выполняют запись только при `confirm=true`;
- удаление — **мягкое** (пометка на удаление), физического `DELETE` нет;
- после правки `.env` **перезапустите Claude Desktop**.

---

## Памятка по безопасности

- Пароль — только в `.env` (он в `.gitignore`); **не** вписывать в `claude_desktop_config.json`.
- На стороне 1С включать в «Состав OData» лишь нужные объекты.
- По умолчанию сервер **read-only** (`READ_ONLY=true`); запись включается осознанно (см. выше).
