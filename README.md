# 1c-odata-mcp

MCP-сервер для доступа к данным **1С:Предприятие** через стандартный интерфейс **OData** — только на чтение. Превращает технический OData в набор понятных инструментов для **руководителя** и **бухгалтера**: контрагенты, документы, остатки, дебиторка, продажи, движение денег.

Работает локально (или на вашем сервере), подключается к **Claude Desktop** и **Claude Code**. Никаких компонентов на стороне 1С ставить не нужно — достаточно опубликованного OData.

> **Статус:** 🚧 в активной разработке. Ниже отмечено, что уже готово, а что в работе. README обновляется по мере реализации.

---

## Зачем это нужно

Сырой OData 1С — это сотни технических `EntitySet` с кириллическими именами (`Catalog_Контрагенты`, `Document_РеализацияТоваровУслуг`, `AccumulationRegister_ТоварыНаСкладах`) и GUID-ключами. Работать с ним из чата невозможно.

Этот сервер прячет всю технику за высокоуровневыми инструментами. Вы спрашиваете обычным языком:

- «Покажи дебиторку» → сальдо счёта 62 из регистра бухгалтерии
- «История по контрагенту Ромашка» → все документы + взаиморасчёты
- «Остатки на складе» → виртуальная таблица Balance регистра накопления
- «Продажи за май» → обороты по реализациям

---

## Безопасность: только чтение

На первом этапе **любые операции записи запрещены**. Защита в три контура:

1. **HTTP-клиент** делает только `GET`; гард `assertReadOnly()` блокирует прочие методы.
2. **Конфигурация** `READ_ONLY=true` (менять на 1-м этапе нельзя).
3. **На стороне 1С** в «Состав OData» включаются только нужные объекты — даже теоретически сервер не видит лишнего.

Секреты — только из `.env`, в логи не попадают (заголовок `Authorization` редактируется). Пароль в репозиторий не коммитится.

## Настройка OData в 1С на Scloud

1. Войдите в базу в режиме **«Предприятие»**.
2. **Все функции → Обработки → «Настройка стандартного интерфейса OData»**.
3. На вкладке **«Состав»** отметьте объекты для публикации (справочники Контрагенты, Номенклатура, Организации; нужные документы; регистры). На 1-м этапе достаточно объектов на чтение.
4. Сохраните.

Адрес OData на Scloud имеет вид:

```
https://terminal.scloud.ru/01/<ваша_база>/odata/standard.odata/
```

Авторизация — **Basic Auth**, учётные данные совпадают с облачной учёткой Scloud (требуется «сквозная авторизация»). Рекомендуется завести **отдельного пользователя только на чтение**.

Проверка вручную (вернётся список объектов в JSON):

```bash
curl -u "ПОЛЬЗОВАТЕЛЬ:ПАРОЛЬ" \
  "https://terminal.scloud.ru/01/<ваша_база>/odata/standard.odata/?$format=json"
```

## Конфигурация (`.env`)

Скопируйте `.env.example` в `.env` и заполните:

```bash
cp .env.example .env
```

| Переменная | Назначение | По умолчанию |
|---|---|---|
| `ODATA_BASE_URL` | Базовый URL OData (со слэшем на конце) | — |
| `ODATA_USERNAME` / `ODATA_PASSWORD` | Basic Auth | — |
| `ODATA_TIMEOUT_MS` | Таймаут запроса, мс | `30000` |
| `ODATA_RETRIES` | Повторы при 429/5xx | `3` |
| `ODATA_PAGE_SIZE` | Размер страницы (`$top`) | `100` |
| `ODATA_MAX_ROWS` | Защитный максимум строк за вызов | `1000` |
| `LOG_LEVEL` | Уровень логов (в stderr) | `info` |
| `READ_ONLY` | Жёсткий режим чтения | `true` |

## Запуск

```bash
npm start
```

Сервер общается по протоколу MCP через **stdio**. Обычно его не запускают руками — его поднимает клиент (Claude Desktop / Claude Code). Команда выше полезна для проверки, что всё стартует без ошибок (логи — в stderr).

## Подключение к Claude Desktop

Откройте конфиг:

- **macOS:** `~/Library/Application Support/Claude/claude_desktop_config.json`
- **Windows:** `%APPDATA%\Claude\claude_desktop_config.json`

Добавьте сервер (укажите абсолютный путь к собранному `dist/index.js`):

```json
{
  "mcpServers": {
    "1c-odata": {
      "command": "node",
      "args": ["/Users/<вы>/Projects/1c-odata-mcp/dist/index.js"],
      "env": {
        "ODATA_BASE_URL": "https://terminal.scloud.ru/01/<ваша_база>/odata/standard.odata/",
        "ODATA_USERNAME": "your_user",
        "ODATA_PASSWORD": "your_password"
      }
    }
  }
}
```

Перезапустите Claude Desktop. В списке инструментов появится `1c-odata`.

> Переменные можно держать и в `.env` рядом с проектом — тогда блок `env` в конфиге не нужен.

## Инструменты

> 🚧 Реализация в процессе. Набор и сигнатуры могут уточняться.

| Инструмент | Что делает | Статус |
|---|---|---|
| `list_entities` | Карта объектов базы по классам (справочники/документы/регистры) | ⏳ |
| `describe_entity` | Поля и связи конкретного объекта (из `$metadata`) | ⏳ |
| `find_counterparty` | Поиск контрагента по названию или ИНН | ⏳ |
| `get_counterparty` | Карточка контрагента + договоры/счета | ⏳ |
| `search_documents` | Поиск документов по типу/дате/контрагенту/сумме | ⏳ |
| `get_document` | Документ с табличной частью | ⏳ |
| `get_inventory` | Остатки товаров по складам | ⏳ |
| `get_debtors` | Дебиторская задолженность (сальдо сч. 62) | ⏳ |
| `get_sales` | Продажи/выручка за период | ⏳ |
| `get_cashflow` | Движение денежных средств (банк + касса) | ⏳ |
| `get_customer_history` | История взаиморасчётов с покупателем | ⏳ |
| `get_supplier_history` | История взаиморасчётов с поставщиком | ⏳ |
| `health_check` | Проверка соединения и авторизации | ⏳ |

## Архитектура

```
src/
  index.ts            # точка входа
  mcp/
    server.ts         # инициализация MCP SDK, регистрация инструментов, stdio
  odata/
    client.ts         # fetch-обёртка: Basic-auth, таймауты, retry, логирование
    query.ts          # типобезопасный билдер $filter/$select/$top/$skip/$expand
    pagination.ts     # автопрокрутка страниц до лимита
    metadata.ts       # загрузка и парсинг $metadata (EDMX → карта сущностей)
    errors.ts         # классификация и нормализация ошибок
  tools/
    meta.ts           # list_entities, describe_entity, health_check
    counterparties.ts # find/get_counterparty, customer/supplier_history
    documents.ts      # search_documents, get_document
    registers.ts      # get_inventory, get_debtors, get_sales, get_cashflow
  types/
    odata.ts          # типы EDMX и ответов OData
    domain.ts         # доменные типы (Counterparty, Document, Balance…)
  config/
    env.ts            # чтение и валидация .env (zod)
    mapping.ts        # маппинг «человеческое имя ↔ объект 1С / счёт»
```

**Стек:** Node.js 20+, TypeScript (strict), официальный `@modelcontextprotocol/sdk`, нативный `fetch`, `zod` (валидация), `pino` (логи в stderr), `fast-xml-parser` (разбор EDMX).

**Целевая конфигурация:** 1С:Бухгалтерия предприятия 3.0. Имена объектов определяются автоматически из `$metadata` при первом запуске; маппинг подстраивается под фактическую базу.
